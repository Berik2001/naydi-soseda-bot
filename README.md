# 🏠 Roommate Bot — поиск сожителей в Казахстане

Telegram-бот для поиска сожителей: заполняешь анкету или объявление о жилье, листаешь подходящих людей по городу и получаешь **взаимные мэтчи** (как в дейтинге, но для жилья).

**Стек:** Python 3.9+ · [aiogram 3.x](https://docs.aiogram.dev) · PostgreSQL (Supabase) · asyncpg · Redis (опц.) · python-dotenv

---

## ✨ Возможности

- Две роли, задаются на шаге «цель»:
  - 🔍 **Ищущий** (seeker) — нет жилья, ищет куда подселиться (личная анкета);
  - 🏠 **Сдающий** (provider) — есть жильё, ищет соседа (объявление с фото квартиры).
- Анкета через **FSM**: пол → имя → цель → город → район → бюджет/цена → фото → «о себе»/описание. По окончании анкета сразу сохраняется и показывается с меню действий.
- Подбор по городу: тот же город + совпадение пола (парни с парнями, девушки с девушками) + противоположная роль, без повторов (таблица `views`). Бюджет на подбор не влияет.
- ❤️ Лайк / ⭐ супер-лайк / 👎 пропуск. При лайке человеку **сразу** приходит анкета с кнопками «💬 Перейти к переписке» / «👎 Отказаться».
- 🎉 При взаимности — мэтч обоим: имя другого показывается **кликабельной ссылкой** на профиль (работает даже без username).
- Редактирование анкеты по полям, пауза/возобновление показа.
- ⭐ **Премиум** через Telegram Stars: анкета в топе выдачи + «кто меня лайкнул».

## 💬 Команды

| Команда | Описание |
|---|---|
| `/start` | Начало / регистрация анкеты заново |
| `/profile` | Моя анкета (просмотр и изменение) — единственная в меню бота |
| `/search` | Найти сожителей |
| `/premium` | Экран премиума |
| `/pause` | Скрыть анкету из поиска |
| `/resume` | Снова показывать анкету |
| `/help` | Справка |

> В меню-кнопке Telegram показывается только `/profile`; остальные команды работают при ручном вводе (в т.ч. во время регистрации — не перехватываются FSM).

## 🚀 Запуск

### 1. Зависимости

```bash
python3 -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Настройка окружения

```bash
cp .env.example .env
```

Обязательные переменные:
- `BOT_TOKEN` — токен от [@BotFather](https://t.me/BotFather);
- `DATABASE_URL` — строка подключения к PostgreSQL.

Опциональные (см. комментарии в `.env.example`):
- `REDIS_URL` — FSM-хранилище в Redis (устойчивость к рестартам + масштаб); без него используется память;
- `DB_POOL_MIN_SIZE` / `DB_POOL_MAX_SIZE` — размер пула asyncpg (дефолты 1 / 5);
- `DB_STATEMENT_CACHE_SIZE` — выставь `0` при работе через Transaction-mode пулер;
- `SENTRY_DSN` — трекинг ошибок в Sentry (без него выключен);
- `LOG_LEVEL` — уровень логов (по умолчанию `INFO`);
- `ENVIRONMENT` — тег окружения для Sentry (по умолчанию `production`).

### Наблюдаемость и контейнер

- **Ошибки:** задай `SENTRY_DSN` — исключения (включая глобальный обработчик) уходят в Sentry со стектрейсом и алертами.
- **Docker:** есть `Dockerfile` для переносимости (`docker build -t roommate-bot . && docker run --env-file .env roommate-bot`). На Railway сборка идёт через nixpacks (см. `railway.toml`), Dockerfile там не используется.

### 3. База данных

Таблицы (`users`, `views`, `likes`) и миграции применяются **автоматически** при старте. Миграции версионные: каждая выполняется один раз (таблица `schema_migrations`), под advisory-lock — безопасно при нескольких инстансах.

> **Supabase (важно):** прямой хост `db.<ref>.supabase.co` работает только по IPv6.
> Если у провайдера нет IPv6 — используй строку **Session Pooler** (IPv4):
> `postgresql://postgres.<ref>:<password>@aws-1-<region>.pooler.supabase.com:5432/postgres`.
> Символ `@` в пароле кодируй как `%40`.

### 4. Старт

```bash
python bot.py
```

В логах должно появиться `Бот запущен. Ожидаю сообщения...` и строка про FSM-хранилище (`Redis.` или `в памяти`).

## 🧪 Тесты и CI/CD

```bash
pip install -r requirements-dev.txt
pytest                      # запустить все тесты
ruff check .                # линт
```

- **CI** ([.github/workflows/ci.yml](.github/workflows/ci.yml)): на каждый push/PR — линт, проверка компиляции и `pytest` (с эфемерным PostgreSQL для интеграционных тестов слоя БД).
- **Интеграционные тесты БД** ([tests/test_integration_db.py](tests/test_integration_db.py)) проверяют ядро — правила подбора кандидата и взаимные лайки — против реального Postgres. Локально: `TEST_DATABASE_URL=postgresql://postgres@localhost:5432/postgres pytest -q` (без переменной они пропускаются).
- **CD**: Railway деплоит из ветки `main` и ждёт зелёного CI — сломанный код в прод не уедет.
- Чистая бизнес-логика вынесена в тестируемые модули: [validators.py](validators.py) (парсинг/валидация), [texts.py](texts.py) (сборка карточек + экранирование), [media_flow.py](media_flow.py) (сбор медиа).

## 📈 Масштабирование

Всё включается переменными окружения, без правки кода:

1. **Устойчивый FSM** — задай `REDIS_URL` (Redis/Upstash). Незаполненные анкеты переживают редеплой.
2. **Больше соединений к БД** — при росте нагрузки переключи `DATABASE_URL` на **Transaction-mode пулер** Supabase (порт `6543`), выстави `DB_STATEMENT_CACHE_SIZE=0`, подними `DB_POOL_MAX_SIZE` (напр. `20`) и задай `DB_COMMAND_TIMEOUT` (напр. `30`), чтобы зависший запрос не держал слот пула. Готовый пресет — в `.env.example`.
3. **Несколько инстансов** — возможно (FSM в Redis + миграции под advisory-lock). Следующий шаг для очень большой нагрузки — перейти с long polling на webhooks за балансировщиком.
4. **Anti-flood** — throttling-middleware ([middlewares/throttling.py](middlewares/throttling.py)) отсекает спам-апдейты до хендлеров, защищая пул БД. Пороги — в конструкторе `ThrottlingMiddleware`. In-memory (на один инстанс); при масштабе на несколько инстансов вынеси состояние в Redis.

## 🔒 Безопасность

- `.env` в `.gitignore` и **не** попадает в репозиторий.
- Пользовательский текст (имя, город, «о себе») экранируется перед вставкой в HTML-подписи.
- Если токен/пароль БД/пароль Redis утекли — перевыпусти токен через `/revoke` у @BotFather, смени пароль БД в Supabase и пароль Redis в Upstash, затем обнови соответствующие переменные окружения.
